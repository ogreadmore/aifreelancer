Part 1: What & Why (Client & Manager View)

Quick Story — What will happen

Overview: We will gather your messy, scattered data and turn it into clean, useful dashboards that help you make smart decisions. First, we'll figure out what questions you want to answer. Then, we'll connect to your data sources (like Google Analytics, your CRM, etc.) and pull everything into one central, organized place called a "warehouse." Next, we'll clean and model the data to create reliable metrics. Finally, we'll build a few key dashboards, test them to make sure they're accurate, and show you how to use them.

What each step means (plain):

  • Define Metrics: We agree on the exact definitions for your key numbers (e.g., "What counts as an 'active user'?"). This is our "contract" for what the numbers mean.
  • Connect Sources: We get safe, read-only access to your tools and set up automated pipelines to copy the data out.
  • Build Warehouse: We create a central database. Think of it as a clean, organized library for all your business data.
  • Model Data: We write code that takes the raw, messy data and transforms it into clean, reliable tables (e.g., a single `users` table or a `daily_sales` summary).
  • Build Dashboards: We create the visual reports with charts and graphs that you'll use to see trends and answer questions.
  • Test & Launch: We run checks to prove the numbers are correct and then give you and your team access.

For you — the person doing the work:

  • Write metric contracts; collect safe, read‑only access; set up a clean warehouse.
  • Build dbt models with tests; publish 1–2 dashboards; prove accuracy.
  • Document refresh schedules and owners; create a small runbook.

For the client — what it should feel like:

  • Easy: you approve definitions and grant access; we handle the build.
  • Helpful: dashboards answer priority questions clearly.
  • Safe: PII is minimized and protected; changes are tested.
  • Clear: you see before/after and know when data refreshes.

Step 1 — Metrics & Sources

Newbie guide: Why we start with metrics

What is a metric contract? It's a simple document where we write down the exact definition of each key number before we build anything. For example: "Monthly Active Users (MAU) is the count of unique users who logged in at least once in the last 28 days."

Analogy: It's like agreeing on the rules of a game before you play. If we don't agree on what a "goal" is, we can't keep score. This contract ensures everyone in the company is counting the same way and trusts the numbers.

Step 2 — Warehouse

Newbie guide: What is a data warehouse?

A data warehouse is a central database designed specifically for analytics. It's where we bring all your data together from different sources (like your website, CRM, and payment system) to get a complete picture of the business.

Analogy: It's the clean, organized library for your data. Instead of having books scattered in different rooms, we bring them all to one library, organize them by topic (the schemas: `raw`, `staging`, `marts`), and make them easy to find and use for reports.

Step 4 — dbt Staging

Newbie guide: What is dbt and staging?

dbt is a tool that helps us transform data using simple SQL. It's the "T" (Transform) in our ELT (Extract, Load, Transform) process. Staging models are the first step in this transformation. They are simple models that do basic cleaning, like renaming columns or casting data types, to prepare the raw data for more complex joins later.

Analogy: Staging is like the prep station in a kitchen (the *mise en place*). Before you start cooking the main dish, you wash the vegetables, chop the onions, and measure the spices. Staging models do this prep work for our data, ensuring everything is clean and ready before we start combining it.

Part 2: Implementer's Guide

This section contains the detailed, technical step-by-step procedures for execution.

Initial Setup — Step‑by‑Step (SOP)

0) Goals & Metric Contracts

  • Define “north star” metrics (e.g., MQLs, SQLs, Revenue) and exact SQL definitions.
  • Create a 1‑pager: business questions, required cuts (channel, campaign, owner), refresh frequency.
  • Agree on data retention and privacy (which fields are PII and how they’re handled).

1) Warehouse Setup

Postgres (managed) quick start

-- Create roles
CREATE ROLE analytics_reader NOINHERIT;
CREATE ROLE analytics_writer NOINHERIT;
CREATE USER etl_user WITH PASSWORD '…';
GRANT analytics_writer TO etl_user;

-- Schemas
CREATE SCHEMA raw;  -- unmodified loads
CREATE SCHEMA stg;  -- cleaned
CREATE SCHEMA int;  -- joined
CREATE SCHEMA mart; -- dashboards

-- Least privilege
GRANT USAGE ON SCHEMA raw,stg,int,mart TO analytics_reader;
GRANT SELECT ON ALL TABLES IN SCHEMA mart TO analytics_reader;
ALTER DEFAULT PRIVILEGES IN SCHEMA mart GRANT SELECT ON TABLES TO analytics_reader;

Backup & Ops

  • Enable daily automated backups; 7–30 days retention.
  • Force SSL; rotate credentials quarterly.
  • Tag costs and set monthly budget alerts.

2) Connectors & Ingestion

  • Prefer managed connectors (Fivetran/Hevo/Stitch). Budget option: Airbyte or scheduled CSVs.
  • GA4: export via BigQuery (if available) or use a connector/API pulls for core events.
  • CRM: nightly export of leads/deals/activities; include updated_at for incremental loads.

Airbyte/Airflow‑style schedule (pseudocode)

daily 02:00: extract crm_leads → raw.crm_leads_yyyy_mm_dd
daily 02:15: extract crm_deals  → raw.crm_deals_yyyy_mm_dd
daily 02:30: extract ga4_events → raw.ga4_events_yyyy_mm_dd
daily 03:00: transform dbt build (stg → int → mart)

3) dbt Project Bootstrap

Commands

dbt init client_analytics
cd client_analytics
dbt run-operation generate_model --args '{name: stg_leads, schema: stg}'
dbt run-operation generate_model --args '{name: stg_deals, schema: stg}'
dbt run-operation generate_model --args '{name: int_funnel, schema: int}'
dbt run-operation generate_model --args '{name: mart_kpis, schema: mart}'

Tests & Docs

-- example in schema.yml
models:
  - name: stg_leads
    columns:
      - name: email
        tests: [not_null, unique]
      - name: first_seen_at
        tests: [not_null]
  - name: int_funnel
    columns:
      - name: stage
        tests:
          - accepted_values:
              values: ['lead','mql','sql','won','lost']

4) Dashboard Layer

  • Use Metabase/Looker Studio/Toggle to warehouse. Create “Executive Overview” first.
  • Define filters: date range, channel, campaign, owner. Save default views for weekly review.
  • Enable email pulses each Monday 9am with prior week trends and anomalies.

5) Scheduling & CI

GitHub Actions (example)

name: nightly-analytics
on:
  schedule: [{ cron: '0 3 * * *' }]
jobs:
  run-dbt:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
      - run: pip install dbt-postgres
      - run: dbt deps && dbt build --profiles-dir .
        env:
          DBT_USER: ${{ secrets.DBT_USER }}
          DBT_PASSWORD: ${{ secrets.DBT_PASSWORD }}

6) Data Quality & Alerts

  • dbt tests must pass ≥ 95% before dashboards refresh; failing tests page ops.
  • Freshness checks: raw.* tables updated within 36h; alert if stale.
  • Anomaly detection: simple z‑score on daily KPIs; flag ±2.5σ deviations.

Lean ELT Pipeline

  • Extract via vendor exports or APIs (daily); prefer managed connectors when budget allows.

  • Automate runs on a daily schedule; backfill initial 12–24 months where feasible.

Fast‑Track (7–8 days)

  • Limit sources to CRM + GA4; 90‑day backfill; skip forecast until week 2.
  • Single “Executive Overview” dashboard first; cohorts later.
  • Daily 30‑min check‑ins to remove blockers quickly.

If Connectors Fail

  • Use vendor CSV exports on a temporary S3/GCS bucket + scheduled loads.
  • Clearly mark dashboards with a “data stale” banner until connectors recover.
  • Switch to weekly refresh if nightly costs spike unexpectedly.

Templates

Example SQL — mart_kpis

with daily as (
  select date_trunc('day', first_seen_at) as d, count(*) as leads
  from int.int_funnel where stage = 'lead' group by 1
), mql as (
  select date_trunc('day', first_mql_at) as d, count(*) as mqls
  from int.int_funnel where first_mql_at is not null group by 1
)
select d as date, 'leads' as kpi_name, leads as value from daily
union all
select d, 'mqls', mqls from mql;

Data Dictionary (starter)

email: unique lowercased email address (primary key for leads)
first_seen_at: timestamp when contact first appeared in any source
utm_source/campaign: marketing attribution fields
stage: normalized funnel stage (lead/mql/sql/won/lost)

Metric Definition (example)

Name: Marketing Qualified Lead (MQL)
Definition: Lead with score ≥ 60 OR requested demo.
Owner: Growth
Upstream: stg_leads, int_funnel
Freshness: daily
Used in: Executive Overview, Funnel dashboard
