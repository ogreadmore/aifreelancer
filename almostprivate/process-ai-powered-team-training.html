Part 1: What & Why (Client & Manager View)

Quick Story — What will happen

Overview: We collect your SOPs and important docs into one organized library. We connect that library to an AI assistant that answers questions and shows sources. We set permissions so people only see what they should. We test it with real questions and then roll it out with short training. Each week, we refresh content and improve the assistant using real questions.

What each step means (plain):

  • Inventory & Baseline: list sources; collect example questions; baseline onboarding time and interruptions.
  • Ingestion: sync sources with citations and “last verified” dates.
  • Permissions: set SSO groups; default deny; audit logs.
  • Design: answer style, confidence handoff to a human expert, stale‑content handling.
  • QA: test with a golden set of questions; fix gaps and tone.
  • Rollout & Train: team intro, quick‑start guide, office hours.
  • Operate: weekly refresh, adoption report, gap backlog.

For you — the person doing the work: Treat this as a concrete checklist you can follow:

  • Ask for access to SSO group admin and read access to source‑of‑truth repositories.
  • Export the baseline (onboarding time, interruptions/week, top repeated questions).
  • Build a source map (repositories, owners, freshness targets).
  • Ingest and chunk docs; enable citations; set refresh cadence and 404 monitors.
  • Configure collections & permissions with SSO groups; default deny; audit logs.
  • Create the answer style guide and draft sample answers.
  • Run a golden set QA of 50 questions; fix gaps; owner sign‑offs.
  • Train the team and schedule office hours; operate weekly.

For the client — what it should feel like:

  • Easy: grant access, confirm sources/owners, and approve the style; we handle the rest.
  • Helpful: employees get instant, cited answers; senior staff interruptions drop quickly.
  • Safe: SSO permissions; default deny; audit logs; stale‑content warnings.
  • Clear: monthly adoption and freshness reports show progress.

Communication & Reporting

  • Weekly admin sync: ingestion health, stale content, top queries, gap backlog
  • Office hours (bi‑weekly): live Q&A; collect feedback; share tips
  • Monthly adoption report: MAU, helpfulness, freshness, top sources

Delivery Phases

  1. Discovery: Baseline metrics; example questions; source inventory; owners.
  2. Design: Ingestion plan; permissions map; answer style; handoff rules.
  3. Build: Ingest, chunk, index; citations; freshness fields; dashboards.
  4. QA & UAT: Golden set eval; tone, citations, and freshness fixes; sign‑off.
  5. Launch: Training, quick‑start guides, office hours.
  6. Operate: Weekly refresh, adoption reports, gap backlog.

Stage Gates

  • Design sign‑off: sources, owners, permissions, style
  • QA pass: helpfulness ≥ 4.5/5 on golden set
  • Launch: adoption plan approved; training scheduled

Risks & Mitigations

  • Stale content → freshness fields, owner reminders
  • Access leaks → default deny, SSO groups, audit logs
  • Low adoption → quick wins, office hours, shout‑outs

KPIs & Reporting

  • MAU: monthly active users vs target population (goal ≥ 60%)
  • Helpfulness: 1–5 rating on answers (goal ≥ 4.5)
  • Freshness: % items “last verified” < 60 days (goal ≥ 90%)
  • Interruptions↓: self‑reported senior interruptions down ≥ 40%

Glossary (Newbie‑friendly)

MAU: Monthly Active Users — how many people actually use it.

Citations: Links to the exact part of the SOP used for an answer.

Chunk: A small, labeled piece of a larger document.

Confidence: How sure the assistant is; low confidence triggers handoff.

Freshness: When content was last verified by an owner.

Part 2: Implementer's Guide

This section contains the detailed, technical step-by-step procedures for execution.

Initial Setup (SOP)

  1. Collect example questions; identify top repeat topics.
  2. Confirm source‑of‑truth repositories; assign owners and freshness targets.
  3. Ingest sources; chunk by headings; store URLs.
  4. Enable citations and answer style rules; add “see also” suggestions.
  5. Configure SSO groups and permissions; default deny; audit logs.
  6. Build dashboard for MAU, helpfulness, freshness, and top queries.
  7. Run golden set QA; fix gaps; owner sign‑offs.

Access & Prerequisites

  • SSO access (Okta/Google) with group mapping
  • Read access to Confluence/Notion/Drive and policy repositories
  • Admin for observability (dashboards) and audit logs
  • Owner list with emails; freshness targets per collection

Design Notes

  • Short answers with links; include “last verified” date for trust.
  • Confidence thresholds: “ask expert” when unsure or content is stale.
  • Collections per team/topic; avoid cross‑leakage; default deny.
  • Gap backlog from unanswered queries; route to owners weekly.

Security & Compliance

  • Least privilege; group mapping via SSO; audit all access.
  • Do not use private data for model training; control retention.
  • Redact sensitive fields in logs; handle RTBF requests.
  • Owner approvals for policy content; legal sign‑off if regulated.
