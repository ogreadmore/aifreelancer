Quick Story — What will happen

Overview: We plan the structure, write copy, design components, then build and test. We launch with safety nets: rollback plan, analytics, and performance budgets. After go‑live, we optimize based on user signals.

What each step means (plain):

  • Discovery: agree on goals and who the site/app is for; outline pages and flows.
  • UX & copy: make simple drafts (wireframes) and write clear copy.
  • Build: turn designs into fast, accessible pages/components.
  • Test: check that everything works, loads quickly, and is readable for everyone.
  • Launch: go live with a plan to undo changes if needed; watch for errors.

For you — the person doing the work: checklist to follow:

  • Confirm goals/KPIs, write IA and content plan; set budgets (CWV/a11y).
  • Create wireframes and copy; get sign‑off; freeze changes for QA.
  • Build components/pages; connect CMS/integrations; add analytics.
  • Run QA: a11y, CWV, SEO, cross‑browser; fix blockers.
  • Launch with monitoring and rollback; schedule optimization.

For the client — what it should feel like:

  • Clear: you see the plan and weekly progress.
  • Safe: we test thoroughly and can roll back quickly.
  • Fast: pages feel snappy and pass performance checks.
  • Accessible: everyone can use it comfortably.

Step 1 — Discovery

Newbie guide: Start with the map

What this is: We decide who we’re building for, what they need to do, and how they’ll find it.

  • Write one sentence: “This site helps [audience] do [main task].”
  • List top tasks: learn pricing, book demo, read docs, contact support.
  • Sketch a site map: Home → Product → Pricing → Contact (keep it short).
  • Pick one primary CTA per page (e.g., “Book a demo”).

Why it matters: A simple map avoids confusion later and keeps the build focused.

Acceptance criteria:

  • Primary KPI named and measurable.
  • Baselines saved to the project folder.
  • Site map fits on one page and includes canonical URLs.

Step 2 — UX & Copy

Newbie guide: Boxes before colors

Wireframes: Black‑and‑white boxes showing headings, text, images, and buttons. No fonts or colors yet.

  • Start with mobile. Stack sections in the order they should be read.
  • Each section needs a headline, short text, and a clear action.
  • Avoid clutter: more white space = easier to read.

5‑second test: Show a friend and ask “What’s this page about? What would you do next?” If it’s unclear, rewrite the headline or move the CTA up.

Pro tip: Write benefit‑first headlines. Replace jargon with simple, active verbs.

Acceptance criteria: Wireframes for the 3 key pages are approved; each page has a single primary CTA; copy drafts exist for all hero sections.

Step 3 — Build

Newbie guide: Reusable pieces

Think LEGO. Build blocks once and reuse them to stay consistent and fast.

  • Tokens: Define colors, font sizes, and spacing scale (e.g., 4/8/12px) once.
  • Buttons: Primary/secondary/ghost with hover/focus/disabled states.
  • Forms: Labels above inputs, helpful error messages, visible focus ring.

Pro tip: Document component props and add Storybook/Playroom for quick visual checks.

Newbie guide: Performance budget

Set simple speed limits and build to meet them.

  • Images: Under 200KB above the fold; use AVIF/WebP; set width/height; provide srcset + sizes.
  • Fonts: 1–2 families, 2–3 weights; preload critical, use font-display: swap, subset if large.
  • Scripts: Defer non‑critical JS; remove unused; prefer native features; split by route.
  • Hints: Use fetchpriority and rel="preload" for critical assets; loading="lazy" for below‑fold media.
  • Measure: Run Lighthouse and WebPageTest; fix anything flagged red; re‑test on staging.

Acceptance criteria: No red Lighthouse items; bundle sizes within budget; above‑the‑fold images optimized.

Step 4 — QA

Newbie guide: Test like a user

Keyboard: Press Tab and Shift+Tab to move around. You should see a clear outline (focus) on links and buttons.

Phone: Try on a real phone. Can you read it without zooming? Do buttons feel big enough?

Slow network: In DevTools, throttle to “Slow 3G”. If it feels sluggish, compress images and defer scripts.

Pro tip: Run Axe or Lighthouse a11y checks; test reduced motion and high‑contrast modes; verify form errors are announced to screen readers.

Acceptance criteria: Zero Sev‑1 defects; obvious a11y blockers fixed; pages readable and interactive within target budgets on a budget phone profile.

Step 5 — Launch

Newbie guide: Safe go‑live

Before launch: Run through a launch checklist (DNS/SSL, redirects, sitemaps/robots, backups, monitoring on).

During: Deploy at a calm time. Keep a status page ready.

After: Crawl the site, watch logs, and check CWV snapshots for regressions. Pre‑warm caches.

If something breaks: 1) Revert deploy, 2) validate DNS/CDN, 3) communicate, 4) fix, 5) retry.

Acceptance criteria: No broken critical paths post‑deploy; redirects correct; monitoring green; rollback steps documented and tested.

Step 6 — Optimize & Experiments

Newbie guide: Improve with evidence

Pick one change: e.g., “Shorter headline increases demo bookings.” Keep the test simple.

Measure one thing: Choose the single metric you want to move (e.g., clicks on “Book demo”).

Give it time: Run until you have enough visits to be confident (don’t stop early).

Pro tip: Start on high‑traffic pages and test big, obvious changes first (headline, hero image, CTA).

Acceptance criteria: Each experiment has a hypothesis, owner, start/end, and a decision (ship/stop) recorded.

Communication & Reporting

  • Slack channel: #proj-creative-dev for day‑to‑day. Response target: same business day.
  • Weekly demo (30 min): Three bullets — Shipped, Next, Needs/Blockers. Link to staging + changelog.
  • Decisions and sign‑offs: Brief, IA/wires, copy, tokens/components. Capture approvals in writing (Slack thread or doc comment).
  • Definition of Done: Passes CWV budgets, WCAG 2.2 AA checks, link checker 0 criticals, design tokens applied, analytics firing.
  • Launch report: CWV lab + field snapshot, a11y summary, SEO crawl issues, 404/redirect map, monitoring + rollback status.
  • Escalation: Sev‑1 (outage/regression) = pause feature work, roll back, notify stakeholders, incident note in #proj-creative-dev.

KPIs & Quality

  • Core Web Vitals: LCP < 2.5s; INP < 200ms; CLS < 0.1.
  • Accessibility: WCAG 2.2 AA checks pass; zero critical issues.
  • SEO: Indexable pages; unique titles/meta; internal links; no major crawl errors.
  • Conversion: Baseline → +10–20% in 30–60 days.
  • Error rate: JS errors/pageview trending down.
  • Uptime: ≥ 99.9% with monitoring and alerts.
How to measure
  • CWV: Lighthouse + field data (CrUX/GA4 web‑vitals).
  • A11y: Axe/Lighthouse; manual keyboard/screen reader passes.
  • SEO: Search Console coverage + performance; site crawl.
  • Errors: Sentry or equivalent; alert on spikes.
  • Uptime: External monitor with SMS/email alerts.

Part 2: Implementer's Guide

Detailed, technical procedures for execution.

Glossary (Newbie‑friendly)

CWV: Core Web Vitals (LCP/INP/CLS) — key performance measures.

WCAG: Accessibility standard that ensures usability for all.

IA (Information Architecture): How pages and content are organized and linked.

Design tokens: Named values for color, spacing, fonts to keep designs consistent.

SSR/SSG: Server‑Side Rendering / Static Site Generation (ways to build pages).

CDN: A network of servers that serves your site from locations closer to users.

Hydration/Islands: Techniques to add interactivity without loading too much JS.

Rollback: A plan to revert a deployment if problems appear.

Initial Setup (SOP)

  1. Confirm goals, KPIs, audiences; draft IA and content plan.
  2. Establish budgets (CWV/a11y/SEO); select stack (framework/CMS/hosting).
  3. Create repo; add formatter/linter; set branch protections; enable preview deploys.
  4. Prepare design tokens, component inventory, and analytics plan; add feature flags if needed.
  5. Set up CI checks (lint, type, unit, a11y, link), and basic CWV tests.
  6. Define a freeze window before QA; decide rollback and monitoring; document runbook.

Access & Prerequisites

  • Repo + CI permissions; secrets vault; feature flag dashboard (if used)
  • Design files; copy source; image/DAM access; license tracking sheet
  • Analytics (GA4/Plausible), Search Console, tag manager (if used)
  • Monitoring (Sentry), uptime checks, and access to logs
  • Hosting/DNS control; CDN and SSL (HSTS enabled post‑launch)

Design Notes

  • Images: Use srcset/sizes; include width/height; prefer AVIF/WebP; lazy‑load offscreen.
  • Typography: Limit families/weights; preload critical fonts; font-display: swap; ensure 16px+ base size.
  • Layout stability: Reserve space; avoid CLS from ads/embeds; use aspect‑ratio.
  • Motion & prefs: Respect prefers-reduced-motion; provide alternatives for animated content.
  • Accessibility: Keyboard paths; visible focus; color contrast; clear error messages; labels/aria.
  • SEO: Descriptive titles/meta; canonical; sitemap; schema where relevant.
  • Icons/media: Sprite or inline SVGs; compress; defer non‑critical assets.

Security & QA

  • Secrets & access: Vault storage; least privilege; 2FA; separate envs (dev/stage/prod).
  • Headers: CSP (script/style nonces), HSTS, X‑Content‑Type‑Options, Referrer‑Policy, COOP/COEP as needed.
  • Transport & cookies: HTTPS; secure + HttpOnly + SameSite cookies; CSRF defenses.
  • Supply chain: Dependency scanning; pin versions; SRI on third‑party assets where applicable.
  • QA gates: Log defects by severity; zero Sev‑1 before launch; smoke tests post‑deploy.
