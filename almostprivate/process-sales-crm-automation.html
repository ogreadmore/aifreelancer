Quick Story — What will happen

Overview: We will make sure people who show interest (called "leads") hear back fast. First, we look at how things work today so we have a starting point. Next, we connect the website and forms to the CRM (the place we store lead information). Then we create short automated messages that go out to new leads. After that we test everything in a safe place so real people are not bothered. Finally we turn the system on for real and watch for problems for a couple of days.

What each step means (plain):

  • Export baseline: download current reports so we know what numbers look like now (this is the "before" picture).
  • Give access: the person doing the work needs a safe account to the CRM and messaging tools; this keeps your real accounts safe.
  • Wire intake: connect website forms to the CRM so leads arrive automatically (no copy-paste).
  • Build messages: write 3–6 short replies (email or SMS) that sound friendly and include a calendar link so leads can book time.
  • Test in sandbox: run everything with fake leads to check that messages send and that CRM records update correctly.
  • Launch slowly: turn the system on for a small amount of traffic first, watch dashboards, and fix anything that breaks.

For you — the person doing the work: Think of this as a clear checklist. Do these simple actions in order:

  • Ask for access (give instructions to the client on exactly what to share).
  • Export the baseline and save the CSV file in the project folder.
  • Create fields and rules in the CRM (this decides who gets each lead).
  • Write short message templates and add personalization (like the lead's name).
  • Test end-to-end with a few fake leads so you can see exactly what happens in the CRM and messages.
  • Go live small (25–50% traffic), monitor for 48 hours, then increase to 100% if stable.
  • Report back with a short note that shows the starting numbers and early results.

For the client — what it should feel like:

  • Easy: they only need to grant access and confirm the goals (you do the rest).
  • Fast: new leads get a friendly reply automatically instead of waiting for someone to act.
  • Safe: everything is tested first so real customers are not spammed.
  • Clear: the client gets a short report showing the "before" numbers and early improvements.

Newbie guide: Why we export a baseline first

What is a baseline? Before we make any changes, we export a report of the last 30 days of activity. This is the "baseline."

Analogy: Think of it like taking "before" photos of a room you're about to renovate.

You do this for two simple reasons:

  • To see the transformation: The "before" photos let you prove how much better the room looks afterward. This baseline file does the same—it gives us the starting numbers so we can measure and celebrate the improvements we make.
  • As a "Correction Map": If the renovation goes wrong and you accidentally plaster over a light switch, the "before" photo shows you exactly where to find it. This file does the same for data.

Example: Imagine an automation bug accidentally assigns 50 new leads to the wrong sales team. The baseline file is our list of those 50 people and who their owner *should* have been. We can use that list to manually fix the errors.

To be perfectly clear: This is NOT a system backup. You cannot press a button and "restore" the entire CRM from this file. It is a simple, powerful reference list for targeted corrections.

How to Export the Baseline (Non-Technical Steps):

  1. Open the CRM (HubSpot or Salesforce) and go to Reports.
  2. Find a report called "Contacts with Activities" or "Leads with Activities".
  3. Set the date range to the last 28–30 days (do not include today).
  4. Export → CSV and save it to /01_discovery/baseline/ with a filename that includes the dates (e.g. baseline_2025-07-01_to_2025-07-30.csv).
  5. Upload this file to our shared project folder.

Step 2 — Pipeline & Fields

Newbie guide: Mapping the sales process

What this is: In this step, we're creating a simple map of your sales process and defining the information we need to collect about each lead. Think of it like planning a road trip: we need to know the stops along the way and what information we need for each stop.

  • Pipeline Stages: These are the main stops on your sales journey (e.g., "New Lead," "First Meeting Booked," "Proposal Sent," "Deal Won"). We'll give them clear, simple names.
  • Exit Criteria: For each stop, we'll write down the one single thing that has to happen to move to the next stop. For example, to move from "New Lead" to "First Meeting Booked," the exit criteria is "a meeting is booked on the calendar." This removes all guesswork.
  • Field Map: This is the list of information we need to collect, like a packing list for our trip. It includes basics like email and first_name, and maybe marketing info like utm_source (which tells us where the lead came from, e.g., Google or Facebook).
  • Dedupe Rules: This is our "no duplicates" rule. We'll tell the system to check for an existing person by their email address to avoid creating multiple records for the same person, which can be very confusing.

Why it matters: A clear map prevents confusion. If we don't define the stages and fields clearly, the automation might put leads in the wrong place or lose important information. This step ensures the whole system is built on a solid, logical foundation.

Step 3 — Design Sign‑off

Newbie guide: Agreeing on the plan

What this is: Before we start building, we create a simple, one-page document that summarizes the "rules of the road" for the automation. This is our blueprint. We then send this to the client for their approval.

This blueprint includes:

  • Routing Rules: This decides who gets which lead. For example, a rule might be: "All leads from California go to Jane." Or, we could use a Round-Robin system, which is like dealing cards, where each new lead goes to the next person in line to ensure fairness.
  • Sequence Matrix: This is a simple timeline of the messages we'll send. For example:
    • Day 1, Minute 1: Send Welcome Email.
    • Day 1, Minute 15: Send SMS (if they gave us a phone number).
    • Day 3: Send Follow-up Email with a case study.

Why it matters: Getting the client to say "Yes, this plan looks good" before we build is the most important step to ensure the project is a success. It makes sure we are all on the same page and avoids surprises or having to redo work later. It's like a homeowner approving the architect's blueprint before the construction crew starts building the house.

Step 4 — Build Core Flows

Newbie guide: Connecting the pipes

What this is: This is the "plumbing" phase. We're connecting the different systems together so that data can flow automatically.

  • Intake / Webhook: When someone fills out a form on your website, the website sends out an instant, automatic alert. This alert is called a Webhook. We "catch" this alert using a tool like Zapier. This is the "intake" process—how we get leads into our system.
  • Upsert: This is a fancy word for a simple, smart action. When a new lead comes in, the system checks if that person's email is already in the CRM.
    • If it is, it UPdates their existing record with the new information.
    • If it's not, it inSERTs them as a new contact.
    This is the magic that prevents creating confusing duplicates.
  • Enrichment: This is where we can add extra, useful public information to a lead's record. For example, if a lead signs up with their work email, an enrichment tool can often find their company's size, industry, and location. This helps us understand the lead better without having to ask them 20 questions on the signup form.

Why it matters: This is the core of the automation. Getting this right means that leads arrive in the CRM instantly and reliably, without anyone having to do manual copy-and-paste, which is slow and prone to errors.

Step 5 — Sequences & Routing

Newbie guide: Writing the messages

What this is: Now that the plumbing is connected, we write the actual messages that will be sent automatically. These are called Templates.

  • Tokens: To make the messages feel personal, we use placeholders called Tokens. A token looks like {{first_name}}. When the email is sent, the system replaces the token with the actual lead's data. So, "Hi {{first_name}}," becomes "Hi Jane,".
  • STOP Handling: We must give people an easy way to opt out. We'll test to make sure that if someone replies "STOP" or "unsubscribe" to a message, they are immediately and automatically removed from all future automated messages. This is a legal requirement and also just good manners.

Why it matters: The messages must be friendly, helpful, and on-brand. This step ensures that the automated outreach feels human and makes a great first impression. We also confirm that the opt-out works perfectly to respect people's wishes and comply with anti-spam laws.

Step 6 — Logging & Dashboards

Newbie guide: Building the scoreboard

What this is: In this step, we set up the "scoreboard" (the Dashboard) so we can see if our automation is working and if we're winning. We also make sure every action is recorded.

  • Activity Logging: We'll set up the system so that every time an automated email or SMS is sent, a note is automatically added to the lead's record in the CRM. This note will say something like, "Sequence Email #1 Sent." This is called Activity Logging. It creates a perfect history of every interaction.
  • Dashboard Tiles: A dashboard is a collection of "tiles," where each tile shows a key number. We'll create tiles for our main KPIs, like "Speed-to-Lead" and "Meetings Booked."

Why it matters: You can't improve what you don't measure. This scoreboard tells us at a glance if the automation is doing its job effectively. If Speed-to-Lead is slow, or if not enough meetings are being booked, the dashboard will show us, and we'll know we need to make adjustments.

Step 7 — QA & UAT

Newbie guide: Test driving the system

What this is: This is our "test drive" phase. Before we let real customers interact with the system, we test it thoroughly to make sure everything works perfectly. This process has two parts:

  • QA (Quality Assurance): This is when the technical team runs a series of tests to try and "break" the system on purpose. We'll test things like:
    • What happens if someone submits the same email twice? (It shouldn't create a duplicate).
    • What happens if someone replies "STOP"? (The sequence should end immediately).
    • What happens if a lead comes in that doesn't match any of our routing rules? (It should go to a default person, not get lost).
  • UAT (User Acceptance Testing): This is when we ask the client to do some test drives of their own. They'll get to use the system just like their team would, to make sure it feels right and works as they expect.

Why it matters: This is the most critical step for building a safe and reliable system. Just like you wouldn't want to fly on a plane that hasn't been tested, you don't want to run your customer-facing operations on untested automation. This step finds and fixes problems before they can affect real leads.

Step 8 — Launch & Hypercare

Newbie guide: Going live

What this is: This is the "Go-Live" day, when we turn the automation on for real leads. But we do it very carefully.

  • Rollout Plan: We don't just flip a switch and turn it on for 100% of leads at once. That's too risky. Instead, we do a "phased rollout." We might start by turning it on for just 25% of new leads.
  • Hypercare: For the first 48 hours after we go live, we are in "hypercare" mode. This means we are watching the dashboards and error alerts like a hawk. We're ready to jump in and fix any unexpected issues immediately.

Why it matters: A slow and careful launch is a safe launch. By starting with a small percentage of leads and watching closely, we can ensure everything is running smoothly and fix any small problems before they become big problems. This protects the client's business and reputation.

Step 9 — Ongoing Optimization

Newbie guide: Making it even better

What this is: The project doesn't end at launch. Now we move into a phase of continuous improvement, where we make small, smart tweaks to make the system even better over time.

  • Weekly Review: Every week, we'll spend 15 minutes looking at the dashboard. Are the numbers good? Are there any new errors? It's a quick health check.
  • A/B Tests: This is a simple way to test ideas. We can try two different subject lines for an email and send them to different groups of new leads. The one that gets more opens and replies wins, and we use that one for everyone going forward. It's a scientific way to improve our messaging.

Why it matters: A good automation system is like a garden; it needs regular attention to flourish. By constantly monitoring and running small tests, we can ensure the system continues to produce better and better results for the business month after month.

Glossary (Newbie‑friendly)

Speed‑to‑lead: How fast you contact a new lead (lower is better).

MQL / SQL: Marketing Qualified Lead (meets marketing criteria) / Sales Qualified Lead (sales believes it’s worth a call/demo).

Upsert: Update if the record exists; insert if it doesn’t (prevents duplicates).

Dedupe: Removing or merging duplicate records (often by matching email or company domain).

Sequence: A pre‑planned series of touches (emails/SMS/calls) over days.

Enrichment: Adding missing info (e.g., company size) from a data provider.

Fast lane: High‑priority routing rule (e.g., high lead score) to assign immediately.

SPF/DKIM/DMARC: DNS records that prove your emails are authorized; improves deliverability.

Activity logging: Writing a note/event into the CRM for each automated touch so reporting is accurate.

4) Sequences & Templates

  • Create MQL Sequence (HubSpot Sequences/Outreach/Apollo): 6–8 touches, 10 days, email/SMS/call mix.
  • Use personalization tokens: {{first_name}}, {{company}}, {{last_page_viewed}}.
  • Create calendar link (Calendly/HubSpot Meetings) and add to first two emails.
  • Define reply‑stop words to auto‑stop sequence on booking or negative reply.

5) Ownership & Routing

  • Round‑robin: maintain last_assigned_at; skip out‑of‑office reps.
  • Territory rules: by country/state/domain; add fallback to general queue.
  • High‑intent fast lane: if score ≥ 70 or “booked demo” → assign immediately and alert Slack.

6) Logging & Observability

  • Every automated touch writes an activity/task with fields: channel, template_id, status, error_code.
  • Error inbox: centralize failed actions with retry button; send daily digest.
  • Dashboards: speed‑to‑lead, touches per lead, logging coverage, reply rates.

Design Notes

  • Field mapping: document IDs, types, picklists; freeze changes during build.
  • Routing: define ownership by territory, round‑robin, product, or score.
  • Scoring: start simple (fit + behavior); iterate with lift analysis.
  • Sequences: 6–10 touches, channel mix email/SMS/call, escalation to human.
  • Logging: all automated touches must write activity with template ID and outcome.
  • Sandboxes: test flows with obfuscated data; back up prod config before deploy.

Security, Privacy, Compliance

  • Principle of least privilege; use service accounts where supported.
  • Respect do‑not‑contact lists; include opt‑out links; honor quiet hours.
  • Log consent fields; document processors; align with CAN‑SPAM/GDPR/CCPA as applicable.
  • Rate‑limit outbound; monitor hard bounces and spam complaints.

Audit & Monitoring (Ops)

Operations & verification guidance — copy this section into your runbook

Purpose

Verify the build matches the design, catch regressions early, and keep the automation healthy after launch.

Immediate Audit (run at launch)

  • Compare exported baseline CSV with live post-launch exports (same columns + counts).
  • Run synthetic leads from each source and confirm: arrival, upsert (no dupes), owner routing, activity logging (with template_id), and message delivery.
  • Test opt-out handling (reply STOP / click unsubscribe) and suppression lists.
  • Force enrichment timeout/failure and confirm intake does not block.
  • Verify SPF/DKIM/DMARC and check a delivered inbox for bounces/spam.

Key Metrics to Monitor (suggested tiles)

  • Median Speed‑to‑Lead (24h): target < 2 minutes.
  • Activity Logging Coverage: % new leads with ≥1 activity in 24‑48h — target ≥ 95%.
  • Error Rate: API/automation failures (5m avg) — alert if > 1%.
  • Bounce / Deliverability: hard bounces in 24h — alert if > 2%.
  • Duplicate Rate and Enrichment Failures (daily).

Alerts & Runbook (short)

  • Critical: Speed‑to‑lead median > 5 min for 15m → page on‑call + Slack.
  • High: Activity logging < 90% for two checks → Slack + create ticket.
  • Triage steps: reproduce with synthetic lead → check logs → disable offending template/flow → rollback if needed.

Synthetic Tests (low effort)

Run a scheduled lightweight script every 5–15 minutes that POSTs a test lead (with a fixed test email) for each intake path. Verify the lead appears, activity is logged, and sequence is triggered. Store results in a tiny status file or push metrics to your monitoring system.
